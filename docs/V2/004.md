# V2-Day 4: 肥包围盒与移动预测 (Fat AABB & Movement)

## 1. 目标
解决“高频重构”带来的性能瓶颈。在物理世界中，物体每帧都在微动。如果每帧都更新树，即使是 $O(\log N)$ 也会积少成多。
今天我们将实现：
1.  **AABB 扩展 (Fat AABB)**：给物体穿上一层“肥衣服”，只要物体在衣服里动，就不用更新树。
2.  **移动预测 (Displacement Prediction)**：根据速度预测下一帧位置，进一步减少更新频率。
3.  **代理更新 (MoveProxy)**：实现一套平滑的“删除-再插入”逻辑。

---

## 2. 核心概念

### A. 肥包围盒 (Fat AABB)
*   **普通 AABB**：紧紧包裹物体的形状。
*   **Fat AABB**：在普通 AABB 的基础上，向四周外扩一段距离（常量 `k_aabbExtension`，通常取 0.1 ~ 0.2 单元）。
*   **逻辑**：只有当物体的 **紧包围盒** 穿透了 **肥包围盒** 的边界时，才触发树的节点重构（Remove + Insert）。

### B. 位移预测 (Movement Prediction)
*   如果物体正在高速向右运动，我们不仅要把 AABB 变肥，还要**向右侧额外拉长一段距离**。
*   **目的**：让高速物体在接下来的几帧内都留在肥包围盒里，避免每一帧都重构。

## 3. 必须实现的功能清单

### A. 扩展 `AABB` 结构体
- [ ] **添加 `Contains(const AABB& other)` 函数**：判断一个 AABB 是否完全包含另一个 AABB。
- [ ] **完善 `Combine(const AABB& other)`**：确保合并逻辑正确。

### B. 实现 `DynamicTree::MoveProxy` 函数
- [ ] **签名设计**：`bool MoveProxy(int32_t proxyId, const AABB& aabb, const Vector2& displacement)`。
- [ ] **包含性判定**：
    *   检查传入的 `aabb` 是否被该节点现有的 `m_nodes[proxyId].aabb`（即 Fat AABB）包裹。
    *   如果包裹且位移很小，直接返回 `false`（表示节点未移动，无需宽相更新）。
- [ ] **重构逻辑**：
    *   如果需要更新：调用 `RemoveLeaf(proxyId)` 将其从旧位置切除。
    *   **扩展 AABB**：将传入的 `aabb` 加上扩展常数，并根据 `displacement` 预测位移。
    *   调用 `InsertLeaf(proxyId)` 将其插入新位置。
- [ ] **返回结果**：返回 `true` 告知宽相（Broadphase），该物体需要重新进行碰撞检查。

### C. 设置扩展常量
- [ ] 定义 `k_aabbExtension = 0.1f` (单位长度)。
- [ ] 定义 `k_aabbMultiplier = 2.0f` (位移预测倍率)。

## 4. 技术实现细节 (Tips)

### 1. 肥包围盒的计算公式
```cpp
// 示例代码片段
AABB fatAABB;
Vector2 extend(k_aabbExtension, k_aabbExtension);
fatAABB.min = aabb.min - extend;
fatAABB.max = aabb.max + extend;

// 位移预测（displacement 是速度 * dt）
if (displacement.x < 0.0f) fatAABB.min.x += k_aabbMultiplier * displacement.x;
else fatAABB.max.x += k_aabbMultiplier * displacement.x;

if (displacement.y < 0.0f) fatAABB.min.y += k_aabbMultiplier * displacement.y;
else fatAABB.max.y += k_aabbMultiplier * displacement.y;
```

### 2. 为什么不直接在 `Update` 里改坐标？
千万不能直接修改树中节点的 AABB 坐标而不重构！因为 AABB 的变化会破坏树的 SAH 启发式布局和自平衡属性。**必须执行 Remove -> 修改坐标 -> Re-Insert。**

### 3. 性能开关
`MoveProxy` 的返回值是宽相优化的关键。只有返回 `true` 的物体，才会被放入“待查队列（Move Buffer）”，从而避免对静止物体进行重复的碰撞检测。

## 5. 今日验收测试 (Unit Test)

1.  **微动不更新测试**：
    *   创建一个 Proxy，记录其初始 Fat AABB。
    *   移动物体 `0.01` 距离，调用 `MoveProxy`。
    *   **预期**：`MoveProxy` 返回 `false`，节点在树中的位置（Slot 索引和 Parent）保持不变。
2.  **大跨度更新测试**：
    *   移动物体 `5.0` 距离（超出 `k_aabbExtension`）。
    *   **预期**：`MoveProxy` 返回 `true`，节点的父节点可能发生变化，树的高度可能通过 `Balance` 调整。
3.  **预测方向测试**：
    *   向 `(+5, 0)` 方向移动物体。
    *   **预期**：更新后的 Fat AABB 在 `max.x` 方向的冗余空间明显大于 `min.x` 方向。
